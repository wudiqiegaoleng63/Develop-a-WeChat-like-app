### 建表
所有表都是用软删除的方式，软删除的好处是：通过一个DeletedAt字段修改是否删除，而不是调用delete删除该表项，维护表中索引的花销要更低一些。

#### UserInfo
```go

	Id            int64          `gorm:"column:id;primaryKey;comment:自增id"`
	Uuid          string         `gorm:"column:uuid;uniqueIndex;type:char(20);comment:用户唯一id"`
	Nickname      string         `gorm:"column:nickname;type:varchar(20);not null;comment:昵称"`
	Telephone     string         `gorm:"column:telephone;index;not null;type:char(11);comment:电话"`
	Email         string         `gorm:"column:email;type:char(30);comment:邮箱"`

	Gender        int8           `gorm:"column:gender;comment:性别，0.男，1.女"`
	Signature     string         `gorm:"column:signature;type:varchar(100);comment:个性签名"`
	Password      string         `gorm:"column:password;type:char(18);not null;comment:密码"`
	Birthday      string         `gorm:"column:birthday;type:char(8);comment:生日"`
	CreatedAt     time.Time      `gorm:"column:created_at;index;type:datetime;not null;comment:创建时间"`
	DeletedAt     gorm.DeletedAt `gorm:"column:deleted_at;type:datetime;comment:删除时间"`
	IsAdmin       int8           `gorm:"column:is_admin;not null;comment:是否是管理员，0.不是，1.是"`

```

Uuid是唯一索引，用于唯一识别userinfo。

Telephone是普通索引，之所以不是唯一索引是因为软删除可能会删除某个Telephone，但该表项还保留在数据表中，再次用该Telephone新建时如果是唯一索引，就无法新建，所以也是用普通索引的方式来创建的。

gorm.DeletedAt是gorm框架里专门用于软删除的结构体。

```go
// soft_delete.go
type DeletedAt sql.NullTime

// sql.go
// NullTime represents a time.Time that may be null.
// NullTime implements the Scanner interface so
// it can be used as a scan destination, similar to NullString.
type NullTime struct {
	Time  time.Time
	Valid bool // Valid is true if Time is not NULL
}
```

根据注释可以看出，NullTime中的Time就是time中的Time结构体，还有一个Valid，用于判断该空时间是否有效。如果无效，说明当前时间为空，即还没删除；如果有效，说明当前时间不为空，已经被删除了。

### 日志库
#### logger
一个好的日志库能够很好地被应用在各种项目需求中，这里我们要开发一个日志库，为了让它能够用在更多的情况下，它需要具备以下的能力：

> 1. 支持把日志写入到多个输出流中，比如测试开发环境到控制台和日志文件，生产环境到日志文件。
> 2. 多级别日志等级。常见有INFO，WARN，ERROR，FATAL，DEBUG，TRACE等。
> 3. 支持结构化输出。比如JSON格式输出，方便日志传输交互。
> 4. 支持日志切割。按照日期，文件大小等对日志进行切割。
> 5. 辅助信息。除了业务信息之外，还要有定位信息，包括函数，行号，时间等。
>

go语言有原生的logger，但是它有一些弊端：

> 1. 只有Print/Println，不支持INFO，DEBUG等多日志级别。
> 2. 错误日志只提供FATAL，PANIC，没有ERROR。
> 3. 缺乏结构化日志格式能力。
> 4. 无日志切割能力。
>

#### zap logger
相比于logrus，zap的并发处理能力更高。其中包括它每个要写入的字段都得携带类型，不需要通过反射来获得字段类型。

**自定义zap logger**

自定义的zap logger能够提供JSON结构化，多输出流，多日志级别等功能。下面是比较基础的自定义zap logger，录友们可以根据自己的需求实现自己的zap logger。

```go
// 自动调用
func init() {
	encoderConfig := zap.NewProductionEncoderConfig()
	// 设置日志记录中时间格式
	encoderConfig.EncodeTime = zapcore.ISO8601TimeEncoder
	// 日志encoder还是JSONEncoder，把日志行格式化成JSON格式
	encoder := zapcore.NewJSONEncoder(encoderConfig)
	conf := config.GetConfig()
	logPath = conf.LogPath
	file, _ := os.OpenFile(logPath, os.O_CREATE|os.O_APPEND|os.O_WRONLY, 644)
	fileWriteSyncer := zapcore.AddSync(file)
	core := zapcore.NewTee(
		zapcore.NewCore(encoder, zapcore.AddSync(os.Stdout), zapcore.DebugLevel),
		zapcore.NewCore(encoder, fileWriteSyncer, zapcore.DebugLevel),
	)
	logger = zap.New(core)
}

// getCallerInfoForLog 获得调用方的日志信息，包括函数名，文件名，行号
func getCallerInfoForLog() (callerFields []zap.Field) {
	pc, file, line, ok := runtime.Caller(2) // 回溯两层，拿到写日志的调用方的函数信息
	if !ok {
		return
	}
	funcName := runtime.FuncForPC(pc).Name()
	funcName = path.Base(funcName) // Base函数返回路径的最后一个元素，只保留函数名

	callerFields = append(callerFields, zap.String("func", funcName), zap.String("file", file), zap.Int("line", line))
	return
}

func Info(message string, fields ...zap.Field) {
	callerFields := getCallerInfoForLog()
	fields = append(fields, callerFields...)
	logger.Info(message, fields...)
}

func Warn(message string, fields ...zap.Field) {
	callerFields := getCallerInfoForLog()
	fields = append(fields, callerFields...)
	logger.Warn(message, fields...)
}

func Error(message string, fields ...zap.Field) {
	callerFields := getCallerInfoForLog()
	fields = append(fields, callerFields...)
	logger.Error(message, fields...)
}

func Fatal(message string, fields ...zap.Field) {
	callerFields := getCallerInfoForLog()
	fields = append(fields, callerFields...)
	logger.Fatal(message, fields...)
}

func Debug(message string, fields ...zap.Field) {
	callerFields := getCallerInfoForLog()
	fields = append(fields, callerFields...)
	logger.Debug(message, fields...)
}
```

init是go中包级别的初始化函数，不需要显示调用，每个包可以包含任意数量的init函数，这些函数在包被导入时自动执行，处于包级别变量声明和main函数执行之间执行。

### http response格式
```go
// controller.go
func JsonBack(c *gin.Context, message string, ret int, data interface{}) {
	if ret == 0 {
		if data != nil {
			c.JSON(http.StatusOK, gin.H{
				"code":    200,
				"message": message,
				"data":    data,
			})
		} else {
			c.JSON(http.StatusOK, gin.H{
				"code":    200,
				"message": message,
			})
		}
	} else if ret == -2 {
		c.JSON(http.StatusOK, gin.H{
			"code":    400,
			"message": message,
		})
	} else if ret == -1 {
		c.JSON(http.StatusOK, gin.H{
			"code":    500,
			"message": message,
		})
	}
}

```

为了方便前后端联合调试，这里所有的网络状态码返回结果都是http.StatusOK即200。通过gin框架中gin.H结构体返回具体的code和message。前端按照相应的code将message在f12控制台打印或者回显到页面中。

ret = 0时是业务走完正常流程返回的结果码，状态码统一为200；ret = -1是系统错误，比如序列化失败，redis缓存失败等，状态码统一为500；ret = -2是业务数据问题导致未正常走完业务流程返回的结果码，状态码统一为400。

三层架构优点：

1. 开发人员只需关注其中一层
2. 可维护性高，可扩展性高，可降低层间依赖
3. 有利于标准化

三层架构缺点：

1. 降低系统性能，中间层延迟导致访问数据减慢
2. 导致级联修改，表示层增加一个功能，相应的业务逻辑层和数据访问层也需要增加相应的功能
3. 增加开发成本

**gochat后端使用了三层架构对项目进行开发，一个功能的调用过程如下：**

前端发送请求 -> controller层调用相关api -> service层执行内部业务逻辑 -> dao层通过gorm框架访问后台数据库 -> service层业务处理完成后返回结果 -> controller层调用JsonBack向前端返回结果 -> 前端拿到结果后页面变化或消息回显等。

（本项目在dao层没有与service层完全解耦，准确的三层架构应该是需要单独提供一个dao层，当service层有一个访问后台数据库的新功能时，之前的dao层函数无法提供该服务时，需要重新在dao层创建一个该service层函数对应的dao层后台数据库访问函数，以提供访问服务）

```go
// KafkaInit 初始化kafka
func (k *kafkaService) KafkaInit() {
    //k.CreateTopic()
    kafkaConfig := myconfig.GetConfig().KafkaConfig
    k.ChatWriter = &kafka.Writer{
        Addr:                   kafka.TCP(kafkaConfig.HostPort),
        Topic:                  kafkaConfig.ChatTopic,
        Balancer:               &kafka.Hash{},
        WriteTimeout:           kafkaConfig.Timeout * time.Second,
        RequiredAcks:           kafka.RequireNone,
        AllowAutoTopicCreation: false,
    }
    k.ChatReader = kafka.NewReader(kafka.ReaderConfig{
        Brokers:        []string{kafkaConfig.HostPort},
        Topic:          kafkaConfig.ChatTopic,
        CommitInterval: kafkaConfig.Timeout * time.Second,
        GroupID:        "chat",
        StartOffset:    kafka.LastOffset,
    })
}
```

这里对ChatWriter进行了初始化，重点是Topic，并定义写超时时间为1s，并定义生产者发送消息后不等待任何来自服务器的确认。最后取消自动创建topic，创建topic可以手动创建。

对ChatReader进行初始化，brokers集群按照config中的配置，同时定义topic和读超时，GroupId设为chat，最后每次读都是从新的消息开始，防止如果之前发送了一条错误消息导致服务器出错，之后一直读该消息。这种LastOffset就相当于是读过的就不用再读，只读一遍。

```go
const (
    RequireNone RequiredAcks = 0
    RequireOne  RequiredAcks = 1
    RequireAll  RequiredAcks = -1
)
```

这是对kafka生产者的配置。

1. **<font style="background-color:rgb(253, 253, 254);">R</font>****<font style="background-color:rgb(253, 253, 254);">equireNone (值为0)</font>**<font style="background-color:rgb(253, 253, 254);">:</font>

<font style="background-color:rgb(253, 253, 254);">生产者发送消息后不等待任何来自服务器的确认。</font>

<font style="background-color:rgb(253, 253, 254);">这种方式提供了最低的延迟，但不保证消息已被成功写入 Kafka。</font>

2. **<font style="background-color:rgb(253, 253, 254);">RequireOne (值为1)</font>**<font style="background-color:rgb(253, 253, 254);">:</font>

<font style="background-color:rgb(253, 253, 254);">生产者等待至少一个副本（通常是领导者副本）成功写入消息后的确认。</font>

<font style="background-color:rgb(253, 253, 254);">这种方式提供了较好的平衡，既保证了消息至少被一个副本持久化，又不会引入过多的延迟。</font>

3. **<font style="background-color:rgb(253, 253, 254);">RequireAll (值为-1)</font>**<font style="background-color:rgb(253, 253, 254);">:</font>

<font style="background-color:rgb(253, 253, 254);">生产者等待所有副本（ISR 列表中的副本）都成功写入消息后的确认。</font>

<font style="background-color:rgb(253, 253, 254);">这种方式提供了最高的持久性保证，但可能会增加延迟，因为需要等待更多的副本确认。</font>

```go
// CreateTopic 创建topic
func (k *kafkaService) CreateTopic() {
    // 如果已经有topic了，就不创建了
    kafkaConfig := myconfig.GetConfig().KafkaConfig

    chatTopic := kafkaConfig.ChatTopic

    // 连接至任意kafka节点
    var err error
    k.KafkaConn, err = kafka.Dial("tcp", kafkaConfig.HostPort)
    if err != nil {
        zlog.Error(err.Error())
    }

    topicConfigs := []kafka.TopicConfig{
        {
            Topic:             chatTopic,
            NumPartitions:     kafkaConfig.Partition,
            ReplicationFactor: kafkaConfig.ReplicationFactor,
        },
    }

    // 创建topic
    if err = k.KafkaConn.CreateTopics(topicConfigs...); err != nil {
        zlog.Error(err.Error())
    }

}
```

```go
// user_info_service.go
// Login 登录
func (u *userInfoService) Login(loginReq request.LoginRequest) (string, *respond.LoginRespond, int) {
    password := loginReq.Password
    var user model.UserInfo
    res := dao.GormDB.First(&user, "telephone = ?", loginReq.Telephone)
    if res.Error != nil {
        if errors.Is(res.Error, gorm.ErrRecordNotFound) {
            message := "用户不存在，请注册"
            zlog.Error(message)
            return message, nil, -2
        }
        zlog.Error(res.Error.Error())
        return constants.SYSTEM_ERROR, nil, -1
    }
    if user.Password != password {
        message := "密码不正确，请重试"
        zlog.Error(message)
        return message, nil, -2
    }

    loginRsp := &respond.LoginRespond{
        Uuid:      user.Uuid,
        Telephone: user.Telephone,
        Nickname:  user.Nickname,
        Email:     user.Email,
        Avatar:    user.Avatar,
        Gender:    user.Gender,
        Birthday:  user.Birthday,
        Signature: user.Signature,
        IsAdmin:   user.IsAdmin,
        Status:    user.Status,
    }
    year, month, day := user.CreatedAt.Date()
    loginRsp.CreatedAt = fmt.Sprintf("%d.%d.%d", year, month, day)

    return "登陆成功", loginRsp, 0
}
```

```go
func (u *userInfoService) checkTelephoneExist(telephone string) (string, int) {
    var user model.UserInfo
    // gorm默认排除软删除，所以翻译过来的select语句是SELECT * FROM `user_info` WHERE telephone = '18089596095' AND `user_info`.`deleted_at` IS NULL ORDER BY `user_info`.`id` LIMIT 1
    if res := dao.GormDB.Where("telephone = ?", telephone).First(&user); res.Error != nil {
        if errors.Is(res.Error, gorm.ErrRecordNotFound) {
            zlog.Info("该电话不存在，可以注册")
            return "", 0
        }
        zlog.Error(res.Error.Error())
        return constants.SYSTEM_ERROR, -1
    }
    zlog.Info("该电话已经存在，注册失败")
    return "该电话已经存在，注册失败", -2
}
```

这里补充一点，gorm框架大多数后台数据库访问方法都是默认不会查询到已经软删除的数据的，就是说如果该表项通过DeletedAt软删除了，在查询该表项的时候默认是查不到的，不需要再次在条件里面标明"deleted_at is null"。如果想要数据软删除之后依旧可以访问到，则需要加上.Unscoped()，这样查询到的数据是不会从DeletedAt字段筛选的。

为什么要说这一点，因为后面管理员功能需要知道哪些用户是被删除过的，所以需要查询已经被软删除的数据。

```go
		var user model.UserInfo
		if res := dao.GormDB.Where("uuid = ?", contactId).Find(&user); res.Error != nil {
			zlog.Error(res.Error.Error())
		}
		if user.Status == user_status_enum.DISABLE {
			zlog.Error("用户已被禁用")
			return "用户已被禁用", -2
		}
		contactApply.Status = contact_apply_status_enum.AGREE
		if res := dao.GormDB.Save(&contactApply); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		newContact := model.UserContact{
			UserId:      ownerId,
			ContactId:   contactId,
			ContactType: contact_type_enum.USER,     // 用户
			Status:      contact_status_enum.NORMAL, // 正常
			CreatedAt:   time.Now(),
			UpdateAt:    time.Now(),
		}
		if res := dao.GormDB.Create(&newContact); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		anotherContact := model.UserContact{
			UserId:      contactId,
			ContactId:   ownerId,
			ContactType: contact_type_enum.USER,     // 用户
			Status:      contact_status_enum.NORMAL, // 正常
			CreatedAt:   newContact.CreatedAt,
			UpdateAt:    newContact.UpdateAt,
		}
		if res := dao.GormDB.Create(&anotherContact); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		return "已添加该联系人", 0
	} else {
		var group model.GroupInfo
		if res := dao.GormDB.Where("uuid = ?", ownerId).Find(&group); res.Error != nil {
			zlog.Error(res.Error.Error())
		}
		if group.Status == group_status_enum.DISABLE {
			zlog.Error("群聊已被禁用")
			return "群聊已被禁用", -2
		}
		contactApply.Status = contact_apply_status_enum.AGREE
		if res := dao.GormDB.Save(&contactApply); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		// 群聊就只用创建一个UserContact，因为一个UserContact足以表达双方的状态
		newContact := model.UserContact{
			UserId:      contactId,
			ContactId:   ownerId,
			ContactType: contact_type_enum.GROUP,    // 用户
			Status:      contact_status_enum.NORMAL, // 正常
			CreatedAt:   time.Now(),
			UpdateAt:    time.Now(),
		}
		if res := dao.GormDB.Create(&newContact); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		var members []string
		if err := json.Unmarshal(group.Members, &members); err != nil {
			zlog.Error(err.Error())
			return constants.SYSTEM_ERROR, -1
		}
		members = append(members, contactId)
		group.MemberCnt = len(members)
		group.Members, _ = json.Marshal(members)
		if res := dao.GormDB.Save(&group); res.Error != nil {
			zlog.Error(res.Error.Error())
			return constants.SYSTEM_ERROR, -1
		}
		return "已通过加群申请", 0
	}

}
```

通过联系人申请，参数中ownerId如果是用户就是用户id，如果是群聊就是群聊id，不是群主id。意思是该用户或者该群聊通过contactId的申请。如果是用户，先通过UserInfo表查询出该用户，看是否已经被禁用了。如果未被禁用，就将申请状态改为同意。然后添加对应的UserContact记录。

如果是群聊的话，先通过GroupInfo表查询出该群聊，看是否已经被禁用了。如果未被禁用，就将申请状态改为同意，并添加对应的UserContact记录，还需要向群组中添加该成员，群人数+1。

该api接口用于用户在申请添加联系人时，如果该联系人是用户，则等待用户处理，用户点击同意即申请通过；如果该联系人是群聊，如果是直接入群，则会在申请加群之后，直接调用该api接口，如果是审核的话，也会等群主点击同意后申请通过。

```go
// ParseMultipartForm parses a request body as multipart/form-data.
// The whole request body is parsed and up to a total of maxMemory bytes of
// its file parts are stored in memory, with the remainder stored on
// disk in temporary files.
// ParseMultipartForm calls ParseForm if necessary.
// If ParseForm returns an error, ParseMultipartForm returns it but also
// continues parsing the request body.
// After one call to ParseMultipartForm, subsequent calls have no effect.
func (r *Request) ParseMultipartForm(maxMemory int64) error
```

在POST请求中，有一个enctype属性，规定了form表单在发送到服务器时的编码方式，它有3种值：

1. **application/x-www-form-urlencoded**：默认的编码方式。但是在用文本的传输和MP3等大型文件的时候，使用这种编码就显得 效率低下。
2. **multipart/form-data**：指定传输数据为二进制类型，比如图片、mp3、文件。
3. **text/plain**：纯文体的传输。空格转换为 “+” 加号，但不对特殊字符编码。

那么在传输文件时，post请求会按照Content-Type: multipart/form-data的请求内容格式来请求数据。

而gin框架中提供了ParseMultipartForm函数来解析请求体中的multipart/form-data数据。它规定最多可以存储maxMemory大小的数据到内存中，剩下的会存储到磁盘中。

在ParseMultipartForm函数中，如果成功解析，会将内容存储到对应的MultipartForm结构体中，大家感兴趣的可以去阅读一下ParseMultipartForm的源码。

```go
// MultipartForm is the parsed multipart form, including file uploads.
// This field is only available after ParseMultipartForm is called.
// The HTTP client ignores MultipartForm and uses Body instead.
MultipartForm *multipart.Form

// multipart.Form
// Form is a parsed multipart form.
// Its File parts are stored either in memory or on disk,
// and are accessible via the *FileHeader's Open method.
// Its Value parts are stored as strings.
// Both are keyed by field name.
type Form struct {
	Value map[string][]string
	File  map[string][]*FileHeader
}

// A FileHeader describes a file part of a multipart request.
type FileHeader struct {
	Filename string
	Header   textproto.MIMEHeader
	Size     int64

	content   []byte
	tmpfile   string
	tmpoff    int64
	tmpshared bool
}

// FormFile returns the first file for the provided form key.
// FormFile calls ParseMultipartForm and ParseForm if necessary.
func (r *Request) FormFile(key string) (multipart.File, *multipart.FileHeader, error)

// File is an interface to access the file part of a multipart message.
// Its contents may be either stored in memory or on disk.
// If stored on disk, the File's underlying concrete type will be an *os.File.
type File interface {
	io.Reader
	io.ReaderAt
	io.Seeker
	io.Closer
}
```

在MultipartForm结构体中，有一个multipart.Form的指针，Form中有两个字段，一个Value，一个是File字段。其中File字段是一个map。每个key对应了一个请求文件头。这个请求文件头里比较重要的几个值是Filename文件名，Size文件大小，content文件头内容。我们拿到这些内容就可以清楚文件的基本信息了。

之后将需要文件的key传入FormFile，能够成功读取到对应的File和FileHeader。File提供了几个文件流用于文件的读取。



当前端传来了multipart/form-data数据时，我们先把post的中该数据解析到MultipartForm结构体中，通过文件头的key读取对应的文件流，文件大小，文件名等基本信息。还需要规范文件名。项目中保留了文件后缀，把原文件名规定成avatar_加上ownerId（用户id）来区分头像。之后在指定静态文件路径创造空文件，并将File中的内容拷贝到空文件中。至此，文件上传的后端部分就完成了。最后需要提醒前端“文件上传完成”。

先看看client的结构体，首先是websocket连接指针，当前端跟client一一对应，通过websocket传输message。其次是Uuid，即当前前端用户的用户id。然后是SendTo通道，该通道作为消息的缓冲通道，在Server的转发通道满的时候，会先存放到SendTo缓冲通道中，等到检查到Server的转发通道不满时，将SendTo通道中的message存入到转发通道中。最后是SendBack通道，这是Server将message通过SendBack发送给client。



另外需要给client开2个go协程，1个Read协程，1个Write协程。

```go
// 读取websocket消息并发送给send通道
func (c *Client) Read() {
    zlog.Info("ws read goroutine start")
    for {
        _, jsonMessage, err := c.Conn.ReadMessage() // 阻塞状态
        if err != nil {
            zlog.Error(err.Error())
            return // 直接断开websocket
        } else {
            var message = request.ChatMessageRequest{}
            if err := json.Unmarshal(jsonMessage, &message); err != nil {
                zlog.Error(err.Error())
            }
            log.Println("接受到消息为: ", jsonMessage)
            if messageMode == "channel" {
                // 如果server的转发channel没满，先把sendto中的给transmit
                for len(ChatServer.Transmit) < constants.CHANNEL_SIZE && len(c.SendTo) > 0 {
                    sendToMessage := <-c.SendTo
                    ChatServer.SendMessageToTransmit(sendToMessage)
                }
                // 如果server没满，sendto空了，直接给server的transmit
                if len(ChatServer.Transmit) < constants.CHANNEL_SIZE {
                    ChatServer.SendMessageToTransmit(jsonMessage)
                } else if len(c.SendTo) < constants.CHANNEL_SIZE {
                    // 如果server满了，直接塞sendto
                    c.SendTo <- jsonMessage
                } else {
                    // 否则考虑加宽channel size，或者使用kafka
                    if err := c.Conn.WriteMessage(websocket.TextMessage, []byte("由于目前同一时间过多用户发送消息，消息发送失败，请稍后重试")); err != nil {
                        zlog.Error(err.Error())
                    }
                }
            }
        }
    }
}
```

前端通过websocket把用户所要发送的message读取出来，通过SendTo通道转发给Server。这里有个先后判定逻辑。首先是判断Transmit转发通道是否已经满了，如果没有满，先把SendTo通道的消息，塞给Transmit通道，直到Transmit通道满了。如果SendTo通道中本来没有消息，则在一开始直接塞给Transmit通道；如果Transmit通道是满的，则先塞给SendTo通道，等到后面Transmit通道不满了再从SendTo通道取出来；如果SendTo通道也满了，则说明这一时刻消息发送太多了，服务器传输不过来了，此时会通过client向前端发送一条“由于目前同一时间过多用户发送消息，消息发送失败，请稍后重试”的提示消息。

```go
func (c *Client) Write() {

    for messageBack := range c.SendBack { // 阻塞状态
        // 通过 WebSocket 发送消息
        err := c.Conn.WriteMessage(websocket.TextMessage, messageBack.Message)
        if err != nil {
            zlog.Error(err.Error())
            return // 直接断开websocket
        }

            zlog.Error(res.Error.Error())
        }
    }
}
```

前端通过建立起的websocket连接，向后端对应的client传输要发送的message。后端的client接收到之后，会向server的转发通道转发message。server通过select不断监听通道，当转发通道有message发送过来，server会开始处理message，分析message的类型，以及接收联系人。

首先所有的message转发到server转发通道时，会将对应的message存储到后台Message表中。

如果message是文本类型，若发送给用户，则需要向对应的client的SendBack通道发送message，并且还有向自己的client也发回这条message。这是因为我们发送消息的同时，对方要接收到该消息并在前端更新聊天界面，自己的聊天界面同样也需要更新聊天界面，所以也要向自己的client发送这条消息，做一个消息回显；

若发送给群聊，就需要通过receiveId到GroupInfo表中查询对应的群组，获取他的群成员列表。遍历群成员列表，对每一个群成员列表都上面对单个用户相同的操作，即向对应的client发送消息，最后也需要向自己的client发送消息。

如果message是文件类型，除了存储message时需要使用到不同的字段，以及在打包序列化消息时消息字段也不同之外，其他的流程都和文本类型是一样的。

如果message是通话类型，由于message消息涉及类似于http握手，即自己这端发送start_call消息，对端接收到之后发送receive_call消息。所以只会向对端client发送start_call消息，就不再需要向己端发送start_call消息（不再是为了回显），否则就会报错（后面讲到音视频通话功能的时候会再详细说）

对于离线消息，在Transmit转发通道接收到message时，会事先存储到Message表中。在之后用户登陆的时候，前端会调用/message/getMessageList的api，这时候会把整个聊天记录都返回给前端。而对于在线用户，转发时会有如下判断：

```go
if receiveClient, ok := s.Clients[message.ReceiveId]; ok {
    receiveClient.SendBack <- messageBack // 向client.Send发送

```

如果此时Server中的Clients map中有该用户，说明该用户此时是在线的，这是不需要通过前端调用/message/getMessageList来获取全部的聊天记录。只需要通过SendBack通道把message转发给Client，Client通过websocket传输给前端。前端拿到之后会将message显示出来，这样节省了一次调用后台数据库的操作。如果Server中的Clients map没有该用户，说明该用户不在线，就不会向该用户发送message，而是保存到后台。



#### kafka
如果后端服务器同一时间发送的消息太多太多了，多个client用channel发送给server的消息太多太多了，导致server的转发通道满了，client的SendTo通道也满了，而且此时还一直在发送消息。

这是一种情况就是会出现“由于目前同一时间过多用户发送消息，消息发送失败，请稍后重试”这样的消息返回给前端。但是这样的结果就是想发送的消息因为服务器太繁忙没法发送。

另一种情况就是在client和server之间加入消息队列，比如kafka这样的流量削峰的消息队列，确保client能够在高峰期顺利地将message发送给server。

那么用kafka替换channel的后端服务器结构就出现了。

```go
// 读取websocket消息并发送给send通道
func (c *Client) Read() {
    zlog.Info("ws read goroutine start")
    for {
        // 阻塞有一定隐患，因为下面要处理缓冲的逻辑，但是可以先不做优化，问题不大
        _, jsonMessage, err := c.Conn.ReadMessage() // 阻塞状态
        if err != nil {
            zlog.Error(err.Error())
            return // 直接断开websocket
        } else {
            var message = request.ChatMessageRequest{}
            if err := json.Unmarshal(jsonMessage, &message); err != nil {
                zlog.Error(err.Error())
            }
            log.Println("接受到消息为: ", jsonMessage)
            if messageMode == "channel" {
                // 如果server的转发channel没满，先把sendto中的给transmit
                for len(ChatServer.Transmit) < constants.CHANNEL_SIZE && len(c.SendTo) > 0 {
                    sendToMessage := <-c.SendTo
                    ChatServer.SendMessageToTransmit(sendToMessage)
                }
                // 如果server没满，sendto空了，直接给server的transmit
                if len(ChatServer.Transmit) < constants.CHANNEL_SIZE {
                    ChatServer.SendMessageToTransmit(jsonMessage)
                } else if len(c.SendTo) < constants.CHANNEL_SIZE {
                    // 如果server满了，直接塞sendto
                    c.SendTo <- jsonMessage
                } else {
                    // 否则考虑加宽channel size，或者使用kafka
                    if err := c.Conn.WriteMessage(websocket.TextMessage, []byte("由于目前同一时间过多用户发送消息，消息发送失败，请稍后重试")); err != nil {
                        zlog.Error(err.Error())
                    }
                }
            } else {
                if err := myKafka.KafkaService.ChatWriter.WriteMessages(ctx, kafka.Message{
                    Key:   []byte(strconv.Itoa(config.GetConfig().KafkaConfig.Partition)),
                    Value: jsonMessage,
                }); err != nil {
                    zlog.Error(err.Error())
                }
                zlog.Info("已发送消息：" + string(jsonMessage))
            }
        }
    }
}
```

```go
             if err != nil {

             }
          }

       case client := <-k.Logout:
          {
             k.mutex.Lock()
             delete(k.Clients, client.Uuid)
             k.mutex.Unlock()
             zlog.Info(fmt.Sprintf("用户%s退出登录\n", client.Uuid))
             if err := client.Conn.WriteMessage(websocket.TextMessage, []byte("已退出登录")); err != nil {

             }
          }
       }
    }
}
```

在server中，登录和登出由于触发的次数没这么多，所以没有使用kafka，依旧是channel进行message传输的。而聊天chat使用一个协程去监听kafka，作为消费者。当监听到chat主题有消息了，执行相应的message存储以及转发message流程。



下面是加上kafka的message转发流程。

当前端传给client相关的message后，client需要按照对应主题把message写到kafka。然后server从kafka上读取对应主题的message，并且判断message的类型以及接收联系人，将message再通过channel传到对应client上，client通过websocket传回前端。

server从kafka读取到消息后，对消息的操作其实是和channel版相同的，唯一不同就是消息传入是通过channel传入还是通过kafka传入。

    - <font style="color:rgb(64, 64, 64);">接收到ICE候选后，调用 </font>`<font style="color:rgb(64, 64, 64);">addIceCandidate</font>`<font style="color:rgb(64, 64, 64);"> 将其添加到 </font>`<font style="color:rgb(64, 64, 64);">RTCPeerConnection</font>`<font style="color:rgb(64, 64, 64);"> 中。</font>
11. **<font style="color:rgb(64, 64, 64);">建立P2P连接</font>**<font style="color:rgb(64, 64, 64);">：</font>
    - <font style="color:rgb(64, 64, 64);">当双方客户端交换完ICE候选并成功建立连接后，P2P通道就建立起来了。</font>
    - <font style="color:rgb(64, 64, 64);">此时，双方可以通过这个通道进行实时音视频通信。</font>
