# 教学文档 01: 项目架构与三层架构

### 正确的开发顺序

```
Model → DAO → Service → Controller → 路由注册

① Model层: 定义数据结构（对应03-数据库模型.md）
② DAO层: 数据库连接与操作（对应04-DAO层连接数据库.md）
③ Service层: 业务逻辑实现（对应07-08文档）
④ Controller层: HTTP请求处理（对应08文档）
⑤ 路由注册: 在HTTP服务器中注册路由（对应09文档）

配置管理(02)、日志系统(05)、常量枚举(06)是基础设施，按需穿插使用。
```

## 一、三层架构概念

### 什么是三层架构？

三层架构将整个业务应用划分成：
- **Controller层（表示层）** - 处理HTTP请求
- **Service层（业务逻辑层）** - 执行业务逻辑
- **DAO层（数据访问层）** - 访问数据库

### 请求流程

```
前端请求 → Controller(参数验证) → Service(业务逻辑) → DAO(数据库操作) 
         → Service(处理结果) → Controller(JsonBack返回) → 前端响应
```

### 三层架构优缺点

**优点:**
- 职责分离，开发人员只需关注其中一层
- 可维护性高，可扩展性高
- 有利于标准化

**缺点:**
- 降低系统性能（中间层延迟）
- 级联修改（修改一层可能影响其他层）
- 增加开发成本

## 三、各层职责详解

### Controller层（api/v1/）

**职责:**
- 注册HTTP路由
- 接收前端请求
- 参数绑定和验证
- 调用Service层
- 返回响应给前端

**示例代码结构:**
```go
// user_info_controller.go
func Login(c *gin.Context) {
    var req request.LoginRequest
    if err := c.BindJSON(&req); err != nil {
        JsonBack(c, "系统错误", -1, nil)
        return
    }
    // 调用Service层
    message, userInfo, ret := service.UserInfoService.Login(req)
    // 返回响应
    JsonBack(c, message, ret, userInfo)
}
```

### Service层（internal/service/）

**职责:**
- 实现具体业务逻辑
- 调用DAO层访问数据库
- 调用Redis/Kafka等外部服务
- 处理业务数据转换

**示例代码结构:**
```go
// user_info_service.go
func (u *userInfoService) Login(loginReq request.LoginRequest) (string, *respond.LoginRespond, int) {
    // 1. 调用DAO层查询用户
    var user model.UserInfo
    res := dao.GormDB.First(&user, "telephone = ?", loginReq.Telephone)
    
    // 2. 业务逻辑判断
    if res.Error == gorm.ErrRecordNotFound {
        return "用户不存在，请注册", nil, -2
    }
    if user.Password != loginReq.Password {
        return "密码不正确", nil, -2
    }
    
    // 3. 构建返回数据
    loginRsp := &respond.LoginRespond{
        Uuid:      user.Uuid,
        Nickname:  user.Nickname,
        // ...
    }
    return "登陆成功", loginRsp, 0
}
```

### DAO层（internal/dao/）

**职责:**
- 初始化数据库连接
- 提供GormDB全局实例
- 所有数据库操作都通过这个实例

**示例代码结构:**
```go
// gorm.go
var GormDB *gorm.DB

func init() {
    // 连接数据库
    dsn := "user:password@tcp(host:port)/database?charset=utf8mb4&parseTime=True"
    GormDB, err := gorm.Open(mysql.Open(dsn), &gorm.Config{})
    
    // 自动迁移表结构
    GormDB.AutoMigrate(&model.UserInfo{})
}
```

## 五、软删除概念

所有表都使用软删除方式，详细定义见 **03-数据库模型.md §五**。

**简要说明:**

软删除不是真正删除记录，而是通过 `gorm.DeletedAt` 字段标记：

```go
type UserInfo struct {
    DeletedAt gorm.DeletedAt `gorm:"column:deleted_at"`
}
```

**核心要点:**

| 状态 | DeletedAt.Valid | deleted_at字段 | GORM查询 |
|------|----------------|---------------|---------|
| 未删除 | false | NULL | 可查到 |
| 已删除 | true | 有值 | 自动过滤 |

**为什么Telephone用普通索引？**

软删除后记录仍存在，唯一索引会阻止重新注册：

```go
Telephone string `gorm:"uniqueIndex"` // ❌ 软删除后无法重新注册
Telephone string `gorm:"index"`       // ✓ 软删除后可以重新注册
```

## 六、常量与枚举

项目使用统一的常量定义和枚举类型，详细定义见 **06-constants与枚举.md**。

**常量包 (pkg/constants/):**
- `CHANNEL_SIZE = 100` - WebSocket通道大小
- `SYSTEM_ERROR` - 统一系统错误提示
- `FILE_MAX_SIZE = 50000` - 文件大小限制
- `REDIS_TIMEOUT = 1` - Redis超时时间

**枚举包 (pkg/enum/):**
使用Go的 `iota` 关键字定义，从0自动递增：

| 枚举类型 | 主要值 | 用途 |
|---------|-------|------|
| `contact_status_enum` | NORMAL, BLACK, DELETE | 联系人状态 |
| `contact_type_enum` | USER, GROUP | 联系人类型 |
| `contact_apply_status_enum` | PENDING, AGREE, REFUSE | 申请状态 |
| `add_mode_enum` | DIRECT, AUDIT | 加群方式 |
| `group_status_enum` | NORMAL, DISABLE, DISSOLVE | 群组状态 |
| `message_type_enum` | Text, Voice, File, AudioOrVideo | 消息类型 |
| `user_status_enum` | NORMAL, DISABLE | 用户状态 |

**Why:** 统一的常量和枚举避免魔法数字，提升代码可读性和可维护性。
