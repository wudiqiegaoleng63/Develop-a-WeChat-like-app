## 一、POST /group/updateGroupInfo - 更新群聊信息

### 1.1 接口概述

**功能描述**：群主在群管理中修改群组信息，包括群名称、群公告、加群方式、群头像等。

**核心特点**：
- 支持选择性更新，只更新非空字段
- **关键：同步更新Session表的ReceiveName和Avatar**，确保所有群成员的会话列表显示最新群信息

**业务流程**：
```
前端请求 -> Controller解析 -> Service更新GroupInfo -> Service同步更新Session表 -> 返回结果
```

---

```go
    Avatar   string `json:"avatar"`    // 群头像URL（可选，空则不更新）
```

### 1.6 数据库操作说明

**涉及的数据库表**：

#### GroupInfo表（群组信息表）
| 字段 | 类型 | 说明 |
|------|------|------|
| uuid | char(20) | 群组唯一ID |
| name | varchar(20) | 名称 |
| notice | varchar(500) | 群公告 |
| add_mode | int8 | 加群方式 |
| avatar | char(255) | 群头像URL |

#### Session表（会话表）
| 字段 | 类型 | 说明 |
|------|------|------|
| uuid | char(20) | 会话唯一ID |
| send_id | char(20) | 创建会话人ID（群成员） |
| receive_id | char(20) | 接收会话人ID（群组ID） |
| receive_name | varchar(20) | 会话显示名称（群名称） |
| avatar | char(255) | 会话头像（群头像） |

**操作流程**：
```
1. First(&group, "uuid = ?", req.Uuid)
   -> 从GroupInfo表查询群组信息

2. Save(&group)
   -> 更新GroupInfo表的群组信息

3. Where("receive_id = ?", req.Uuid).Find(&sessionList)
   -> 查询所有以该群组为接收方的会话记录

4. 循环Save(&session)
   -> 更新每条会话记录的ReceiveName和Avatar
```

---

### 1.7 关键概念详解

#### 为什么需要同步更新Session表？

**场景说明**：
- 用户A、B、C都在群组G中
- Session表中存在记录：send_id=A, receive_id=G, receive_name="旧群名"
- 当群主修改群名称为"新群名"时，如果不同步更新Session表
- 用户A的会话列表仍显示"旧群名"，造成数据不一致

**解决方案**：
```go
// 查询所有与该群组相关的会话
Where("receive_id = ?", req.Uuid).Find(&sessionList)

// 批量更新每个会话的显示信息
for _, session := range sessionList {
    session.ReceiveName = group.Name  // 同步最新群名
    session.Avatar = group.Avatar     // 同步最新头像
    dao.GormDB.Save(&session)
}
```

#### 选择性更新策略

**实现原理**：
- 前端只传递需要更新的字段
- 后端通过判断字段值是否为空来决定是否更新
- 避免空值覆盖原有数据

```go
// 使用空字符串判断
if req.Name != "" {
    group.Name = req.Name  // 非空才更新
}

// 使用特殊值判断（int类型）
if req.AddMode != -1 {
    group.AddMode = req.AddMode  // -1表示不更新
}
```

```
4. First(&user, "uuid = ?", member)

5. append(rspList, ...)
   -> 组装响应数据
```

### 2.7 关键概念详解

#### Members字段的JSON存储

**为什么用JSON存储**：
- GORM不直接支持[]string类型
- 使用json.RawMessage类型存储JSON格式的字符串数组
- 数据库中实际存储格式：`["U001", "U002", "U003"]`

**解析过程**：
```go
// GroupInfo模型定义
type GroupInfo struct {
    Members json.RawMessage `gorm:"column:members;type:json"`
}

// 解析JSON为Go切片
var members []string
json.Unmarshal(group.Members, &members)
// 解析后：members = ["U001", "U002", "U003"]
```

#### 遍历查询UserInfo的设计考量

**为什么不一次性查询**：
- 群组成员数量通常较少（几十到几百人）
- 逐个查询可确保每个成员信息的准确性
- 便于错误处理：某个成员查询失败可立即终止

**性能优化建议**：
```go
// 如果群成员较多，可使用批量查询优化：
var users []model.UserInfo
dao.GormDB.Where("uuid IN ?", members).Find(&users)

// 但需要注意结果顺序和数量的一致性
```

## 三、POST /group/removeGroupMembers - 移除群聊成员

### 3.1 接口概述

**功能描述**：群主批量移除群聊成员，执行成员退群的逆向操作。

**核心特点**：
- **不能移除群主**： ownerId校验保护群主不被移除
- **批量删除**：同时删除Members、Session、UserContact、ContactApply记录
- **软删除机制**：使用deleted_at字段标记删除时间

**业务流程**：
```
前端请求 -> Controller解析 -> Service校验群主 -> Service遍历删除成员 -> Service更新GroupInfo -> 清理Redis缓存 -> 返回结果
```

```go
// internal/service/gorm/group_info_service.go

// RemoveGroupMembers 移除群聊成员
func (g *groupInfoService) RemoveGroupMembers(req request.RemoveGroupMembersRequest) (string, int) {
    // 1. 查询群组信息
    var group model.GroupInfo
    if res := dao.GormDB.First(&group, "uuid = ?", req.GroupId); res.Error != nil {
        zlog.Error(res.Error.Error())
        return constants.SYSTEM_ERROR, -1
    }

    // 2. 解析Members字段
    var members []string
    if err := json.Unmarshal(group.Members, &members); err != nil {
        zlog.Error(err.Error())
        return constants.SYSTEM_ERROR, -1
    }

    // 3. 构造软删除时间戳
    var deletedAt gorm.DeletedAt
    deletedAt.Time = time.Now()
    deletedAt.Valid = true
    log.Println(req.UuidList, req.OwnerId)

    // ★4. 遍历待移除成员列表
    for _, uuid := range req.UuidList {
        // ★关键校验：不能移除群主
        if req.OwnerId == uuid {
            return "不能移除群主", -2
        }

        // 5. 从members切片中移除该UUID
        for i, member := range members {
            if member == uuid {
                members = append(members[:i], members[i+1:]...)
            }
        }
        group.MemberCnt -= 1  // 减少成员计数

        // ★6. 删除会话记录（Session表）
        if res := dao.GormDB.Model(&model.Session{}).Where("send_id = ? AND receive_id = ?", uuid, req.GroupId).Update("deleted_at", deletedAt); res.Error != nil {
            zlog.Error(res.Error.Error())
            return constants.SYSTEM_ERROR, -1
        }

        // ★7. 删除联系人记录（UserContact表）
        if res := dao.GormDB.Model(&model.UserContact{}).Where("user_id = ? AND contact_id = ?", uuid, req.GroupId).Update("deleted_at", deletedAt); res.Error != nil {
            zlog.Error(res.Error.Error())
            return constants.SYSTEM_ERROR, -1
        }

        // 8. 删除申请记录（ContactApply表）
        if res := dao.GormDB.Model(&model.ContactApply{}).Where("user_id = ? AND contact_id = ?", uuid, req.GroupId).Update("deleted_at", deletedAt); res.Error != nil {
            zlog.Error(res.Error.Error())
            return constants.SYSTEM_ERROR, -1
        }
    }

    // 9. 更新群组的Members字段
    group.Members, _ = json.Marshal(members)
    if res := dao.GormDB.Save(&group); res.Error != nil {
        zlog.Error(res.Error.Error())
        return constants.SYSTEM_ERROR, -1
    }

    // 10. 清理Redis缓存
    if err := myredis.DelKeysWithPrefix("group_session_list"); err != nil {
        zlog.Error(err.Error())
    }
    if err := myredis.DelKeysWithPrefix("my_joined_group_list"); err != nil {
        zlog.Error(err.Error())
    }

    return "移除群聊成员成功", 0
}
```

**Redis缓存清理说明**：
- **DelKeysWithPrefix**：删除所有以指定前缀开头的Key（不带groupId后缀）
- **group_session_list**：清理群会话列表缓存
- **my_joined_group_list**：清理已加入群列表缓存
- **注意**：不使用`DelKeysWithPattern`（精确匹配），而是使用`DelKeysWithPrefix`（模糊匹配前缀）

### 3.6 数据库操作说明

**涉及的数据库表**：

#### Session表（会话表）
| 删除条件 | 说明 |
|---------|------|
| send_id = uuid AND receive_id = group_id | 删除该成员在该群的会话记录 |

#### UserContact表（联系人表）
| 删除条件 | 说明 |
|---------|------|
| user_id = uuid AND contact_id = group_id | 删除该成员与群的联系人关系 |

#### ContactApply表（申请记录表）
| 删除条件 | 说明 |
|---------|------|
| user_id = uuid AND contact_id = group_id | 删除该成员与群的申请记录 |

**操作流程**：
```
1. First(&group, "uuid = ?", req.GroupId)
   -> 查询群组信息

2. json.Unmarshal(group.Members, &members)
   -> 解析成员列表

3. for _, uuid := range req.UuidList
   -> 遍历每个待移除成员

4. 校验：if req.OwnerId == uuid
   -> 防止移除群主

5. members = append(members[:i], members[i+1:]...)
   -> 从切片中删除成员UUID

6. Update("deleted_at", deletedAt)
   -> 软删除Session、UserContact、ContactApply记录

7. json.Marshal(members)
   -> 将更新后的成员列表转为JSON

8. Save(&group)
   -> 保存群组更新

9. DelKeysWithPattern(...)
   -> 清理Redis缓存
```

### 3.7 关键概念详解

#### 为什么不能移除群主？

**设计原因**：
- 群主是群组的创建者和管理者
- 移除群主会导致群组失去管理权限
- 群主需要通过解散群组来退出（使用DismissGroup接口）

**校验实现**：
```go
// 前端传递owner_id用于校验
for _, uuid := range req.UuidList {
    if req.OwnerId == uuid {
        return "不能移除群主", -2  // 特殊错误码-2
    }
    // ...继续删除其他成员
}
```

#### 批量删除的实现

**删除策略**：
- 移除成员 = 成员退群的逆向操作
- 需要删除：会话(Session)、联系人(UserContact)、申请记录(ContactApply)

**软删除机制**：
```go
// 构造deleted_at时间戳
var deletedAt gorm.DeletedAt
deletedAt.Time = time.Now()
deletedAt.Valid = true

// 使用Update进行软删除（而非Delete）
dao.GormDB.Model(&model.Session{}).
    Where("send_id = ? AND receive_id = ?", uuid, req.GroupId).
    Update("deleted_at", deletedAt)
```

**为什么使用软删除**：
- 保留历史记录，便于数据恢复和审计
- GORM的DeletedAt字段自动过滤已删除记录
- 符合项目的数据管理规范

#### Members切片的动态删除

**Go切片删除元素**：
```go
// 从切片中删除索引i的元素
members = append(members[:i], members[i+1:]...)

// 示例：
// 原切片：["A", "B", "C", "D"]
// 删除索引1的"B"：
// members[:1] = ["A"]
// members[2:] = ["C", "D"]
// append后 = ["A", "C", "D"]
```

#### Redis缓存清理

**为什么需要清理缓存**：
- 群组信息可能缓存在Redis中
- 移除成员后需要清除缓存，确保数据一致性

**清理策略**：
```go
// 使用前缀匹配删除相关缓存（注意方法名）
myredis.DelKeysWithPrefix("group_session_list")       // 群会话列表缓存（不带groupId）
myredis.DelKeysWithPrefix("my_joined_group_list")     // 已加入群列表缓存（不带groupId）
```

**DelKeysWithPrefix vs DelKeysWithPattern 区别**：
- `DelKeysWithPrefix(prefix)`：删除所有以prefix开头的Key，如 `group_session_list_*`
- `DelKeysWithPattern(pattern)`：精确匹配pattern这个Key，不会模糊匹配

## 八、Redis缓存机制

### 8.1 缓存策略概述

本文档三个接口的Redis缓存策略：

| 接口 | 缓存策略 | Redis操作 |
|------|---------|-----------|
| UpdateGroupInfo | 更新时删除 | 读前检查 → 更新DB → 删除缓存 |
| GetGroupMemberList | 读时缓存 | 读缓存 → 未命中则查DB → 写缓存 |
| RemoveGroupMembers | 更新时删除 | 更新DB → 删除相关缓存 |

### 8.2 UpdateGroupInfo缓存流程

```
请求到达
    │
    ├─→ 检查Redis: "group_info_{uuid}"
    │       │
    │       ├─→ 存在 → 删除缓存（确保数据一致）
    │       └─→ 不存在 → 继续
    │
    ├─→ 更新GroupInfo表
    │
    ├─→ 更新Session表
    │
    └─→ 删除缓存: "group_session_list*", "my_joined_group_list*"
```

### 8.3 GetGroupMemberList缓存流程

```
请求到达
    │
    ├─→ 检查Redis: "group_memberlist_{groupId}"
    │       │
    │       ├─→ 存在 → 直接返回缓存数据
    │       │
    │       └─→ 不存在 → 查询数据库
    │               │
    │               ├─→ 查GroupInfo获取Members
    │               ├─→ 遍历查UserInfo获取昵称头像
    │               ├─→ 组装响应数据
    │               └─→ 写入Redis缓存（30分钟过期）
```

### 8.4 RemoveGroupMembers缓存清理

```
移除成员成功
    │
    ├─→ DelKeysWithPrefix("group_session_list")
    │       └─→ 删除所有群会话列表缓存
    │
    └─→ DelKeysWithPrefix("my_joined_group_list")
            └─→ 删除所有已加入群列表缓存
```

### 8.5 Redis方法区别

| 方法 | 用途 | 示例 |
|------|------|------|
| GetKeyNilIsErr(key) | 精确读取单个Key | `GetKeyNilIsErr("group_memberlist_G001")` |
| SetKeyEx(key, value, ttl) | 精确写入单个Key（带过期时间） | `SetKeyEx("group_memberlist_G001", data, 5min)` |
| DelKeyIfExists(key) | 精确删除单个Key（先检查存在） | `DelKeyIfExists("group_info_G001")` |
| DelKeysWithPrefix(prefix) | 模糊删除前缀匹配的所有Key | `DelKeysWithPrefix("group_session_list")` 删除 `group_session_list_*` |
| DelKeysWithPattern(pattern) | 精确删除pattern这个Key | `DelKeysWithPattern("group_info_G001")` 只删除这一个Key |

## 四、总结

### 三接口对比

| 接口 | 核心功能 | 关键操作 |
|------|---------|---------|
| updateGroupInfo | 更新群组信息 | 同步更新Session表的ReceiveName和Avatar |
| getGroupMemberList | 获取成员列表 | 遍历Members查询UserInfo获取昵称头像 |
| removeGroupMembers | 移除成员 | 校验群主、批量删除Session/UserContact、软删除 |

### 设计要点

1. **数据一致性**：更新群信息时同步Session表，确保会话列表显示正确
2. **选择性更新**：空值判断避免覆盖原有数据
3. **遍历查询**：逐个查询UserInfo确保数据准确性
4. **权限保护**：OwnerId校验防止误删群主
5. **软删除机制**：使用deleted_at标记，保留历史记录
6. **缓存管理**：操作后清理Redis缓存保证一致性
