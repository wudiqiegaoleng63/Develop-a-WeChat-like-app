## 一、聊天室功能概述

### 什么是聊天室？

聊天室是一个临时性的多人通信场景：
- 用户可以邀请多个联系人进入聊天室
- 聊天室内的消息可以实时同步
- 聊天室数据存储在内存中（不持久化）

### 聊天室 vs 群聊

| 特性 | 聊天室 | 群聊 |
|------|--------|------|
| 持久化 | 内存存储（不持久化） | 数据库持久化 |
| 成员管理 | 临时性，随时加入/退出 | 固定成员列表 |
| 消息存储 | 不存储历史消息 | 消息表持久存储 |
| 适用场景 | 即时多人讨论 | 长期多人沟通 |

## 五、Controller层完整代码

**源码位置**: `api/v1/chatroom_controller.go`

```go
package v1

import (
    "github.com/gin-gonic/gin"
    "kama_chat_server/internal/dto/request"
    "kama_chat_server/internal/service/gorm"
    "kama_chat_server/pkg/constants"
    "kama_chat_server/pkg/zlog"
    "net/http"
)

// ============================================================
// GetCurContactListInChatRoom - 获取当前聊天室联系人列表
// ============================================================
func GetCurContactListInChatRoom(c *gin.Context) {
    // 1. 参数绑定
    var req request.GetCurContactListInChatRoomRequest
    if err := c.BindJSON(&req); err != nil {
        zlog.Error(err.Error())
        // ★直接返回HTTP响应，不使用JsonBack
        c.JSON(http.StatusOK, gin.H{
            "code":    500,
            "message": constants.SYSTEM_ERROR,
        })
        return
    }
    
    // 2. 调用Service层
    message, rspList, ret := gorm.ChatRoomService.GetCurContactListInChatRoom(req.OwnerId, req.ContactId)
    
    // 3. 返回响应
    JsonBack(c, message, ret, rspList)
}
```

### 关键代码注释

| 代码行 | 说明 |
|--------|------|
| `c.BindJSON(&req)` | 将请求JSON绑定到结构体 |
| `constants.SYSTEM_ERROR` | 使用统一系统错误提示 |
| `gorm.ChatRoomService.GetCurContactListInChatRoom` | 调用Service层获取聊天室列表 |

## 六、Service层完整代码

**源码位置**: `internal/service/gorm/chatroom_service.go`

```go
package gorm

import "kama_chat_server/internal/dto/respond"

// ============================================================
// chatRoomService - 聊天室服务结构
// ============================================================
type chatRoomService struct {
}

// ★全局服务实例
var ChatRoomService = new(chatRoomService)

// ============================================================
// chatRoomKey - 聊天室复合键
// ============================================================
// ★使用复合键（ownerId + contactId）标识一个聊天室
type chatRoomKey struct {
    ownerId   string  // 创建者Uuid
    contactId string  // 联系人Uuid
}

// ============================================================
// chatRooms - 聊天室数据存储
// ============================================================
// ★使用map存储聊天室成员列表
// Key: chatRoomKey复合键
// Value: []string（联系人Uuid列表）
var chatRooms = make(map[chatRoomKey][]string)

// ============================================================
// GetCurContactListInChatRoom - 获取聊天室联系人列表
// ============================================================
func (c *chatRoomService) GetCurContactListInChatRoom(ownerId string, contactId string) (string, []respond.GetCurContactListInChatRoomRespond, int) {
    var rspList []respond.GetCurContactListInChatRoomRespond
    
    // ★从内存map中获取联系人列表
    for _, contactId := range chatRooms[chatRoomKey{ownerId, contactId}] {
        rspList = append(rspList, respond.GetCurContactListInChatRoomRespond{
            ContactId: contactId,
        })
    }
    
    return "获取聊天室联系人列表成功", rspList, 0
}
```

### 数据存储结构

    {ownerId: "U111", contactId: "U222"}: {"U333", "U444", "U555"},

}

## 七、关键技术点

### 7.1 复合键Map

**Why使用复合键struct作为Map Key**:
- Go的map key必须是可比较的类型
- struct只要所有字段都可比较，就可以作为key
- 复合键可以唯一标识"用户A与用户B的聊天室"

```go
// ✅ struct作为map key（推荐）
type chatRoomKey struct {
    ownerId   string
    contactId string
}
var chatRooms = make(map[chatRoomKey][]string)

// ❌ 字符串拼接作为key（不推荐）
var chatRooms = make(map[string][]string)
// key = ownerId + "_" + contactId  // 容易出错
```

### 7.2 内存存储 vs 数据库存储

**Why聊天室使用内存存储**:
- 聊天室数据临时性，不需要持久化
- 内存查询速度快（无数据库IO）
- 适合高频临时通信场景

**How to apply**:
- 需持久化数据（如消息记录）→ 存数据库
- 临时数据（如在线状态）→ 存内存/Redis

## 九、注意事项

### 9.1 并发安全问题

**问题**: `chatRooms` map可能被多协程同时访问

**解决方案**:
```go
// 使用sync.RWMutex保护
type chatRoomService struct {
    mutex *sync.RWMutex
}

func (c *chatRoomService) GetCurContactListInChatRoom(...) {
    c.mutex.RLock()
    // ... 读取chatRooms
    c.mutex.RUnlock()
}
```

### 9.2 内存泄漏问题

**问题**: 聊天室数据一直存在内存中

**解决方案**:
- 定期清理长时间未使用的聊天室
- 使用Redis替代内存存储（支持过期时间）
