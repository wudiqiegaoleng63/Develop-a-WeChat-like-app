## 一、单机架构的瓶颈

### 单机架构问题

```
单机Server:
┌─────────────────┐
│ Gin + WebSocket │
│ MySQL           │
│ Redis           │
│ Kafka           │
└─────────────────┘

问题:
1. 无法水平扩展 → 用户量增长时单机扛不住
2. WebSocket连接数有限 → 单机最多几万连接
3. 单点故障 → Server宕机，全部用户断开
```

### 解决方案：分布式部署

```
分布式架构:
                  ┌─────────┐
                  │ Nginx   │ (负载均衡)
                  └─────────┘
                       │
        ┌──────────────┼──────────────┐
        │              │              │
   ┌─────────┐    ┌─────────┐    ┌─────────┐
   │ Server1 │    │ Server2 │    │ Server3 │
   │ Gin+WS  │    │ Gin+WS  │    │ Gin+WS  │
   └─────────┘    └─────────┘    └─────────┘
        │              │              │
        └──────────────┼──────────────┘
                       │
              ┌────────┼────────┐
              │        │        │
         ┌───────┐ ┌───────┐ ┌───────┐
         │ MySQL │ │ Redis │ │ Kafka │
         │集群   │ │ 集群  │ │ 集群  │
         └───────┘ └───────┘ └───────┘
```

## 二、WebSocket分布式挑战

### 问题：Clients map无法共享

```
Server1的Clients:
┌─────────────────────┐
│ U123 → Client1      │
│ U456 → Client2      │
└─────────────────────┘
Server1无法访问Server2的Clients!

问题:
- ClientA连接Server1
- ClientB连接Server2
- ClientA发消息给ClientB，Server1找不到ClientB
```

### 解决方案：共享Kafka Topic

```
所有Server消费同一个Kafka Topic:

Server1 ────消费────→ chat_message Topic ←────消费──── Server2
                                          ↑
                                     ClientA写入

效果:
- ClientA写入消息到Kafka
- Server1和Server2都能消费
- Server2找到ClientB，转发成功
```

## 三、Kafka消费者组详解

### 消费者组机制

```go
// 当前配置
GroupID: "chat"
```

```
消费者组的工作方式:

同一个消费者组:
- 消息只会被组内一个消费者消费
- 不会重复消费

┌─────────┐        ┌─────────────────┐        ┌─────────┐
│ Server1 │──消费──│ chat_message    │──消费──│ Server2 │
│ Group:chat│      │ Topic           │        │ Group:chat│
└─────────┘        └─────────────────┘        └─────────┘
        ↑                   ↑                   ↑
    消息1               消息2               消息3

消息1 → Server1消费
消息2 → Server2消费
消息3 → Server1消费（轮询）
```

### 本项目的方案

```
方案1: 同一个消费者组
- 消息不重复处理
- 但只能被一个Server处理
- 问题: 如果ClientB在Server2，消息被Server1消费，转发失败

方案2: 不同消费者组（★本项目方案）
- 每个Server用不同的GroupID
- 消息会被所有Server消费
- 每个Server检查自己的Clients，有则转发

┌─────────┐        ┌─────────────────┐        ┌─────────┐
│ Server1 │──消费──│ chat_message    │──消费──│ Server2 │
│ Group:server1│    │ Topic           │        │ Group:server2│
└─────────┘        └─────────────────┘        └─────────┘
        ↑                   ↑                   ↑
    所有消息            所有消息            所有消息

效果:
- 每条消息被所有Server消费
- Server1检查Clients，找到则转发
- Server2检查Clients，找到则转发
```

## 四、修改为分布式方案

### 修改GroupID

```go
// 原来（单机）
GroupID: "chat"

// 分布式（每个Server不同）
// ★方案1: 使用主机名或IP
GroupID: "chat_" + hostname

// ★方案2: 使用随机UUID
GroupID: "chat_" + random.GetRandomString(8)
```

### 转发逻辑修改

```go
// 原来的转发逻辑（有问题）
if receiveClient, ok := s.Clients[message.ReceiveId]; ok {
    receiveClient.SendBack <- messageBack
}

// 分布式转发逻辑（改进）
// ★无论是否找到，都尝试转发
// 找到 → 转发成功
// 未找到 → 其他Server可能找到
if receiveClient, ok := s.Clients[message.ReceiveId]; ok {
    // 本Server有这个用户，转发
    receiveClient.SendBack <- messageBack
} else {
    // 本Server没有，其他Server可能有
    // 消息已存储数据库，用户上线后会拉取
    zlog.Info("用户离线，消息已存储")
}
```

## 五、协程池优化

### 问题：每个消息创建协程

```go
// 原来的做法
func HandleMessage(data []byte) {
    go storeMessage(data)     // 创建协程存储
    go forwardMessage(data)   // 创建协程转发
}

// 问题:
// 10000条消息 = 20000个协程
// 协程创建开销大，资源消耗高
```

### 解决方案：协程池

使用 `ants` 协程池库：

```go
import "github.com/panjf2000/ants/v2"

// 创建协程池
pool, _ := ants.NewPoolWithFunc(100, func(i interface{}) {
    task := i.(*Task)
    task.Execute()
})

// 提交任务（不创建新协程）
pool.Invoke(&Task{
    Func: func() {
        dao.GormDB.Create(&message)
    },
})
```

### 协程池优势

```
协程池工作方式:

┌──────────────────────────────────┐
│ 协程池（固定100个协程）           │
│ ┌────┐ ┌────┐ ┌────┐ ...        │
│ │goroutine│ │goroutine│ │goroutine│          │
│ └────┘ └────┘ └────┘             │
└──────────────────────────────────┘
        ↑
    任务队列

优势:
1. 协程数量固定，防止资源耗尽
2. 协程复用，减少创建开销
3. 任务队列缓冲，流量控制
```

★注意: 这是优化建议，原项目未实现协程池。读者可根据需求决定是否实现。

## 六、分库分表策略

### 为什么需要分库分表？

```
单表问题:
- 用户表：100万用户，查询慢
- 消息表：每天100万消息，存储压力大

解决方案:
- 按用户ID哈希分表
- 每个表存储部分用户数据
```

### 分表策略

```go
// 按用户ID哈希分表
func GetMessageTable(userId string) string {
    // ★取最后一位数字（0-9）
    suffix := userId[len(userId)-1]
    return "message_" + string(suffix)
}

// 查询时
tableName := GetMessageTable(userId)
dao.GormDB.Table(tableName).Find(&messages)
```

### 分表效果

```
message_0  → 用户ID末尾是0的用户消息
message_1  → 用户ID末尾是1的用户消息
...
message_9  → 用户ID末尾是9的用户消息

效果:
- 每个表数据量减少到原来的1/10
- 查询速度提升
```

★注意: 这是优化建议，原项目未实现分库分表。读者可根据数据量决定是否实现。

## 七、Nginx负载均衡配置

### Nginx配置示例

```nginx
upstream chat_servers {
    server 192.168.1.1:8000;
    server 192.168.1.2:8000;
    server 192.168.1.3:8000;
}

server {
    listen 80;
    
    # WebSocket支持
    location /wss {
        proxy_pass http://chat_servers;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
    }
    
    # HTTP API
    location / {
        proxy_pass http://chat_servers;
    }
}
```

### WebSocket负载均衡要点

```nginx
# ★关键配置：支持WebSocket升级
```

## 九、MySQL集群配置

### 主从复制

```
Master（写入） → Slave1（读取）
             → Slave2（读取）

写操作: 只在Master执行
读操作: 可以在Slave执行（减轻Master压力）
```

### GORM配置读写分离

```go
// 使用插件实现读写分离
import "gorm.io/plugin/dbresolver"

// 配置
GormDB.Use(dbresolver.Register(dbresolver.Config{
    // 读操作使用Slave
    Replicas: []gorm.Dialector{
        mysql.Open("slave1_dsn"),
        mysql.Open("slave2_dsn"),
    },
    // 写操作使用Master
    Sources: []gorm.Dialector{
        mysql.Open("master_dsn"),
    },
}))
```
